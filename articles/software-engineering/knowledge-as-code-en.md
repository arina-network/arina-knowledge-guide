# Knowledge as Code. In English. With Math.
_Author [Slava Zemlianskyi](https://www.linkedin.com/in/zemlianskyi/), published 2026-03-27._

## Why understanding should be executable, not just exist

Most organizations treat knowledge like documentation: static, descriptive, and ultimately decaying.

Engineers know a truth: if something matters, it must run.

This leads to a stronger idea:
> Knowledge is not text. Knowledge is code.

Not metaphorically — structurally.

## Knowledge exists only when it acts

Information becomes knowledge only when it drives decisions and actions.

OCA loop (Observation -> Computation -> Action) is the minimal flow. Break the loop, and knowledge collapses:
- Observations without processing -> Noise
- Models without action -> Theory
- Actions without grounding -> Guesswork

If nothing changes in the real world, no knowledge was applied.

## Predictability is a requirement

A piece of knowledge should behave consistently under similar conditions.

If the same situation leads to different decisions with no clear reason, you’re not dealing with knowledge — you’re dealing with instability.

This is why:
- Checklists outperform “experience” in aviation
- Good playbooks outperform heroic decision-making
- Systems outperform individuals

Predictability is what makes knowledge transferable.

## Composition is the multiplier

Isolated insights don’t scale. Connected ones do.

Real systems are built by chaining decisions:
- One step produces an output
- The next step uses it as input

When knowledge is composable:
- Teams can build on each other
- Systems can grow without chaos
- Complexity becomes manageable

When it isn’t:
- Every problem starts from scratch
- Every team reinvents logic
- Organizations stall despite “having smart people”

That’s not a talent issue. That’s a structure issue.

## State - why experience matters

Knowledge doesn’t run in a vacuum — it runs inside a system with memory.

Two people can follow the same process and get different results because they interpret signals differently, prioritize differently, and act differently.

What we call “experience” is accumulated internal state:
- What you notice
- What you ignore
- What you consider risky

So knowledge is not just rules — it’s rules applied to a stateful system.

That’s why copying decisions without context rarely works.

## Uncertainty is not a bug

In real environments, you don’t get certainty — you get likelihoods.

Good knowledge doesn’t eliminate uncertainty. It structures it.

A strong decision system:
- Recognizes multiple possible outcomes
- Assigns confidence to them
- Acts anyway

This is where intuition comes from — not magic, but compressed experience.

People who have seen more patterns make faster probabilistic judgments.

## Every decision has a cost

Applying knowledge is not free. You always pay for:
- Gathering information
- Processing it
- Acting on it

Overthinking is not intelligence — it’s unbounded cost. Underthinking is not speed — it’s unpriced risk.

Good systems don’t maximize “decision quality” in isolation. They optimize the balance between value and cost.

## If it can’t be tested, it isn’t knowledge

If knowledge is executable, it must produce outcomes you can evaluate.

Instead, many organizations:
- Judge people instead of decisions
- Reward confidence instead of accuracy
- Debate opinions instead of measuring results

This leads to a fundamental failure mode: they don’t improve knowledge, they defend it.

Real systems do the opposite:
- Make predictions
- Compare with reality
- Update accordingly

That’s how engineering works. That’s how learning works.

## Knowledge must be versioned

Every meaningful system evolves. Which means knowledge must:
- Change over time
- Be comparable across versions
- Be reversible when wrong

Without versioning:
- Mistakes become permanent
- Improvements are invisible
- Learning is lost

With it:
- You can test alternatives
- You can roll back bad decisions
- You can accumulate progress

This is standard for code - and strangely rare for knowledge.

## Conclusion

If your knowledge:
 -cannot be executed,
- cannot be tested,
- cannot be updated,

then it isn’t really knowledge. It’s documentation. And documentation doesn’t run systems.

Execution does.
